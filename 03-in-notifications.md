> **§3.6** — уведомления.

### 3.6 Уведомления (универсальный модуль 1С → Site)

Единый расширяемый контракт для уведомлений, которые 1С инициирует вручную менеджером или из автоматических бизнес-процессов (пришла оплата, завершается договор, запрос бухгалтерских документов, лист заботы, просрочка договора и т.д.). Сайт принимает одно событие, создаёт запись уведомления на **компанию** и асинхронно **дублирует его в каналы** (in-app, email, SMS, telegram, webpush) по собственным правилам маршрутизации и настройкам пользователей.

**Принципы (обязательны для соблюдения обеими сторонами):**

1. **Адресат — компания, не пользователь.** 1С шлёт уведомление на `login` (код компании в учёте). Сайт сам определяет, кто из пользователей имеет доступ к этой компании, и для каждого из них применяет **его** настройки каналов. 1С не знает и не передаёт список получателей-людей. Конкретное правило fan-out:
  - **владелец индивидуального кабинета** (если `login` принадлежит обычному, не корпоративному пользователю) — он один и есть получатель;
  - **корп. кабинет** (если `login` входит в кабинет): мастер кабинета **всегда** получает уведомление + те сотрудники кабинета, у которых на момент события есть доступ к этой конкретной компании в UI сайта (см. §3.2 «Принципы устройства корпоративного кабинета»). Сотрудники без доступа к компании уведомление **не получают**, даже если они состоят в кабинете.
2. **Тексты не передаются из 1С.** Сайт хранит шаблоны по `notification_code` и рендерит их под канал (у email / SMS / push / in-app разные ограничения и формат). 1С отдаёт только **классификацию** события и **данные** в `params`. Свободные сообщения менеджера — через админку **сайта**, не через этот контракт.
3. **Каналы выбирает сайт**, не 1С. Фактический набор каналов для конкретного уведомления определяется: (а) политикой для `notification_code`, (б) настройками пользователя, (в) комплаенсом (отписки, подтверждение контактов). От 1С приходит только `priority` — он может повышать маршрутизацию.
4. **Ссылочная модель вместо URL.** 1С передаёт идентификаторы сущностей (`contract_id`, `message_id`, `document_id`, `login`), сайт сам строит ссылки в кабинете. Для внешних ссылок (файл из хранилища 1С и т.п.) — отдельное поле `external_url`.
5. **Параметризуемые коды.** Сценарии с величинами (дней до / после, уровень эскалации) — **один** `notification_code` + значения в `params`; новые пороги/уровни не плодят новые коды.
6. **Источник события.** Поле `origin: "manual" | "automated"` (+ опц. `manager_login` при `origin=manual`) — для аудита и аналитики. Свободный текст при `manual` не передаётся.
7. **Приоритет.** Закрытый enum `priority: "low" | "normal" | "high" | "critical"`, default `normal`. Семантика: `critical` — обязательная попытка push + SMS (сайт вправе мягко деградировать по политике каналов); `high`/`normal`/`low` — обычный fan-out.
8. **Fan-out устойчив к ошибкам каналов.** Сайт создаёт **одну** запись уведомления и публикует её в каналы асинхронно; статусы ведутся по каналам отдельно (`pending | sent | failed | retry`). Если у пользователя нет ни одного канала кроме `in_app` — событие считается доставленным (запись in_app создана), остальные каналы помечаются `skipped` с причиной.
9. **Расширение без изменения API.** Новые сценарии — это новые значения `notification_code` в справочнике, а не новые endpoint / event_type. Нетипизированные атрибуты временно живут в `meta`; договор по их промоушену в схему фиксируется отдельным соглашением.
10. **Вне области действия §3.6.** Транзакционные письма (magic-link при `user.create`, invite-email при `corporate.cabinet.create` / `corporate.user.invite` и аналогичные) **не** используют движок §3.6: они отправляются напрямую по email, не подчиняются пользовательским настройкам каналов и политикам отписки (иначе блокируется доступ к кабинету).

#### Enum-ы

`notification_type` (домен события — расширяемо):


| Значение     | Назначение                                          |
| ------------ | --------------------------------------------------- |
| `payment`    | События по оплатам                                  |
| `contract`   | События жизненного цикла договора                   |
| `accounting` | Бухгалтерские документы и сверки                    |
| `mail_care`  | «Лист заботы» — внимание к невостребованным письмам |


`priority` (закрытый enum):


| Значение   | Семантика                                                               |
| ---------- | ----------------------------------------------------------------------- |
| `low`      | Информационное, можно положить в дайджест                               |
| `normal`   | Обычный fan-out по дефолтной политике канала                            |
| `high`     | Приоритетное, доставляется вне дайджеста                                |
| `critical` | Обязательная попытка push + SMS (мягкая деградация — по политике сайта) |


#### `notification.push` — отправить уведомление

Адресат — **компания** по `login`. Сайт сам раздаёт её всем пользователям, у которых есть доступ к этой компании, с учётом индивидуальных настроек каналов (см. принцип 1 выше).

```json
// Запрос от 1С
{
  "event_type": "notification.push",
  "payload": {
    "notification_id":   "8d0215f2-5583-40f9-a130-b45644dbf0d4",   // REQUIRED — UUID, первичный ключ идемпотентности
    "notification_type": "contract",                                // REQUIRED — enum домена
    "notification_code": "contract.expiring",                       // REQUIRED — сценарий внутри домена (см. справочник ниже)
    "login":             "01/02/03",                                // REQUIRED — компания-получатель; fan-out по пользователям — на сайте
    "priority":          "high",                                    // optional — low|normal|high|critical (default normal)
    "origin":            "automated",                               // optional — manual|automated (default automated)
    "contract_id":       "cnt-001",                                 // optional — ссылочные поля по смыслу сценария
    "message_id":        "msg-001",
    "document_id":       "doc-001",
    "external_url":      "https://storage.1c.virtoffice.ua/...",    // optional — когда ссылка действительно внешняя
    "params": {                                                     // REQUIRED — данные для шаблона
      "days_left": 30,
      "end_date":  "2026-06-30"
    },
    "expires_at":        "2026-06-30T23:59:59Z",                    // optional — подсказка UI (авто-скрытие из активных)
    "dedupe_key":        "contract.expiring:cnt-001:2026-06-30",    // optional — fallback идемпотентности
    "meta":              { }                                         // optional — свободные атрибуты
  }
}

// Ответ — принято (создано или идемпотентно)
HTTP 200
{ "message": "Notification accepted.", "notification_id": "8d0215f2-5583-40f9-a130-b45644dbf0d4" }

// Ответ — обновлено (тот же notification_id, изменены params)
HTTP 200
{ "message": "Notification updated.", "notification_id": "8d0215f2-5583-40f9-a130-b45644dbf0d4" }

// Ответ — компания не найдена
HTTP 404
{
  "error":   "company_not_found",
  "message": "Company not found by login",
  "login":   "01/02/03"
}

// Ответ — неизвестный notification_code / ошибка валидации params
HTTP 400
{
  "error":             "invalid_notification_payload",
  "message":           "Invalid notification payload",
  "notification_code": "contract.expiring",
  "fields":            ["params.days_left"]
}
```

> `origin=manual` сохраняется для случая, когда **типизированный** сценарий (например `accounting.docs.requested`) инициирован менеджером в 1С вручную. Свободные тексты в этом контракте **не передаются**.

#### `notification.resolve` — снять активное уведомление

Используется для «живых» уведомлений (закреплённые баннеры, эскалации, листы заботы), когда в 1С зафиксировано, что повод более неактуален (договор продлён, документы получены, письмо забрано и т.д.).

```json
// Запрос от 1С
{
  "event_type": "notification.resolve",
  "payload": {
    "notification_id": "8d0215f2-5583-40f9-a130-b45644dbf0d4",   // REQUIRED (либо dedupe_key)
    "dedupe_key":      "contract.expiring:cnt-001:2026-06-30",   // допускается вместо notification_id
    "resolved_at":     "2026-06-15T10:00:00Z"                   // optional
  }
}

// Ответ — снято
HTTP 200
{ "message": "Notification resolved." }

// Ответ — уже снято / не найдено (идемпотентный успех)
HTTP 200
{ "message": "Notification already resolved or not found." }
```

#### Справочник стартовых `notification_code`


| `notification_code`                  | Описание                                                                                                                                                                                                                                                                          | Обязательные ссылочные поля | `params`                                                                   |
| ------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | --------------------------- | -------------------------------------------------------------------------- |
| `payment.received`                   | Пришла оплата (валюта в системе одна — в контракт сумма не передаётся)                                                                                                                                                                                                            | `login`                     | `document_number`, `document_date`                                         |
| `contract.expiring`                  | Завершается договор (`params.days_left` задаёт сценарий 7/30/60 и т.п.)                                                                                                                                                                                                           | `login`, `contract_id`      | `days_left`, `end_date`                                                    |
| `contract.overdue`                   | Договор просрочен (эскалация через `params.stage`)                                                                                                                                                                                                                                | `login`, `contract_id`      | `stage` (`1`/`2`/`3`...), `days_overdue`, `amount_due`                     |
| `accounting.docs.requested`          | Запрос бухгалтерских документов                                                                                                                                                                                                                                                   | `login`                     | `document_kinds` (array), `period`                                         |
| `accounting.ukrposhta.docs_required` | Укрпочта: требование передать клиенту шаблон договора (закреплённое уведомление со ссылкой на файл; `notification_type = accounting`)                                                                                                                                             | `login`                     | — (ссылка на шаблон — через `external_url`)                                |
| `mail.care_list_uncollected`         | «Лист заботы»: письма не забраны                                                                                                                                                                                                                                                  | `login`                     | `uncollected_count`, `oldest_received_at`                                  |
| `scan_rule.changed_by_manager`       | Менеджер 1С изменил `scan_rule` через `user.update` (§3.1): сайт генерирует этот внутренний код после применения входящего события, если новое значение отличается от предыдущего. Применяется и для первичной установки (когда было `null`), и для последующих правок менеджером | `login`                     | `previous_scan_rule` (`null` при первичной установке), `applied_scan_rule` |

> Справочник расширяется добавлением новых значений `notification_code` + шаблона на стороне сайта; изменение контракта не требуется.

**Паттерн «закреплённого уведомления» (замена прежнего §3.6 Укрпошта):**

- `notification_type = "accounting"`, `notification_code = "accounting.ukrposhta.docs_required"`, `priority = "high"`, `expires_at` не передаётся (живёт до явного `notification.resolve`).
- Ссылка на файл шаблона — `external_url` (например, из хранилища 1С).
- «Одно активное требование на компанию» обеспечивается `dedupe_key = "accounting.ukrposhta.docs_required:{login}"`. Повтор с тем же ключом обновляет `external_url`.
- Снятие уведомления — стандартным `notification.resolve` по тому же `notification_id` или `dedupe_key`.

#### Правила поведения и идемпотентности

- Первичный ключ идемпотентности — `notification_id`; при его отсутствии — `dedupe_key`. Повтор → `200` без повторного fan-out.
- Повтор `notification.push` с тем же `notification_id`, но **изменёнными** `params` — сайт **обновляет** активное уведомление (менеджер / учёт в 1С главенствуют), новое не создаёт.
- `notification.resolve` по уже закрытому или несуществующему уведомлению — `200` без эффекта.
- Ошибка в одном канале fan-out **не** влияет на ответ `200` на входящее событие: доставка по каналам асинхронна и имеет отдельные статусы.

#### Соотношение триггеров с внутренними `notification_code`

Появление данных в кабинете **через `POST /intake/`** как правило не дублируется отдельным `notification.push` из 1С: сайт генерирует внутренний `notification_code` тем же движком §3.6.


| Триггер (IN `event_type` или иной)  | Внутренний `notification_code` (сайт → движок §3.6)                                                                                                                                          |
| ----------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `mail.create`                       | `mail.new`                                                                                                                                                                                   |
| `mail.scan.ready`                   | `mail.scan_ready`                                                                                                                                                                            |
| `mail.status.update`                | внутренний код по конкретному статусу (или no-op, если статус не требует нотификации)                                                                                                        |
| `document.create`                   | `document.new`                                                                                                                                                                               |
| `contract.scan.upload`              | `contract.scan_uploaded`                                                                                                                                                                     |
| `user.update` (поле `scan_rule`)    | `scan_rule.changed_by_manager` (только если правило **изменено** менеджером через IN `user.update`; самостоятельные правки клиента через §4.2 отдельно не нотифицируют других пользователей) |


1С **не** шлёт `notification.push` по этим кодам — это был бы дубль.

Счёт по заявке продления (§3.9) приходит тем же **`document.create`**; подстановка в открытый диалог — по **`extension_request_id`**, fan-out — **`document.new`** как у любого нового документа.
