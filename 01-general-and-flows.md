### 1.2 Аутентификация

Все запросы в обе стороны требуют заголовок:

```http
X-API-Key: {api_key}
```

| Ключ      | Значение                   | Где хранится              |
| --------- | -------------------------- | ------------------------- |
| Site → 1С | `mock-site-to-1c-key-2026` | `settings.ONE_C_API_KEY`  |

Ответ при неверном ключе:

```json
HTTP 401
{
  "error": "Invalid or missing API key"
}
```

### 1.3 Формат запросов и ответов

- Все тела запросов и ответов: `Content-Type: application/json`; кодировка `UTF-8`.
- **Дата-время** (момент во времени, поля `*_at`): только `ISO 8601` в UTC, с суффиксом `Z` → `"2026-03-20T10:00:00Z"`.
- **Календарная дата** (без времени суток, поля `*_date`): только `YYYY-MM-DD` → `"2026-03-20"` (договоры, документы, диапазоны в §4.5).
- **Таймзона бизнеса:** `Europe/Kyiv`. Все времена в API передаются в UTC; локализация в UI — на стороне сайта. Контракт изначально допускает расширение на несколько таймзон: в будущем у компании (`corporate.cabinet.`*) может появиться поле `timezone` (IANA, например `Europe/Kyiv`); до его появления считается дефолт `Europe/Kyiv`.
- **Ограничения по размеру:** тело запроса ≤ `1 MiB`; списочные поля (`companies`, `document_kinds` и т.п.) ≤ `500` элементов. Превышение — `400 payload_too_large`.
- **Rate limiting:** по умолчанию `60 rps` на направление (`1С→Site`, `Site→1С`) и `10 rps` на пару (api-key, event_type / endpoint). Превышение — `429` с заголовком `Retry-After` (секунды). Повторы — по §1.5.

**Семантика PATCH / update-событий:**

- Отсутствующее в теле поле — **не трогать** (обновлением считаются только присутствующие ключи).
- `null` — **явный сброс значения** (очистить поле у объекта на сайте / в 1С).
- Коллекции по умолчанию могут поддерживать **частичный патч**: передаются только изменившиеся/добавленные элементы; связка по стабильному ключу элемента, удаление — явным флагом `"_deleted": true` в элементе. Конкретное событие явно объявляет свою семантику коллекций: либо partial-patch, либо **snapshot** (полная замена), либо **single-purpose** (одно событие = одна сущность, без коллекций). Например: `pricing.upsert.entries[]` — snapshot (§3.8); `params` уведомлений — partial-patch при повторе с тем же `notification_id` (§3.6); компании в кабинете не передаются коллекциями вообще — каждая компания управляется одним событием на одну операцию (`corporate.company.attach / .transfer / .detach`, см. §3.2).

**Единый envelope ошибки** (применяется для всех `4xx`, см. §1.6):

```json
{
  "error":   "Short machine code",
  "message": "Human-readable description (ua/ru/en — на усмотрение стороны)",
  "fields":  ["login"],
  "details": { }
}
```

Поля `fields` и `details` — опциональные; `error` и `message` — обязательные.

**Именование полей контрагента и контактов:**

- `login` — внешний код договора / контрагента в учёте 1С (не email и не учётная запись корп. пользователя). В ответах и ошибках сервера это поле имеет то же имя, что и в запросе.
- Телефон — единое поле `phone` в формате E.164 (`+380991234567`).
- Email — единое поле `email`; дополнительный адрес — `secondary_email`.

**Логин в URL:** должен быть URL-encoded (`01/02/03` → `01%2F02%2F03`). Реализации на обоих концах обязаны корректно декодировать значение перед поиском по нему.

**Адрес сущности** — унифицированный объект `address` для всех мест, где в контракте фигурирует адрес (компания клиента, офис хранения VirtOffice, юридический адрес на договоре и т.п.):

```json
"address": {
  "public": "вул. Хрещатик 1, Київ",
  "exact":  "вул. Хрещатик, буд. 1, оф. 5, м. Київ, 01001"
}
```


| Поле     | Назначение                                                                                                 | Видимость                                            |
| -------- | ---------------------------------------------------------------------------------------------------------- | ---------------------------------------------------- |
| `public` | Сокращённая / обезличенная форма для публичного отображения на сайте (страница контактов, карточки и т.п.) | Видит клиент в UI                                    |
| `exact`  | Полный адрес с деталями: корпус, офис, почтовый индекс, при необходимости — реквизиты                      | Юридические документы — договоры, акты, счета и т.п. |


- Оба поля **опциональные** по отдельности, но при передаче объекта `address` **хотя бы одно** должно быть заполнено; иначе `400 invalid_address`.
- Фоллбэки: если у потребителя нет `public` — допустимо показать `exact` в UI; если нет `exact` — подставить `public` в юр. документ. Политика фоллбэка — на стороне потребителя.
- Семантика патча — общая (см. выше): отсутствующий ключ (`public` или `exact`) не трогается; `null` — сброс значения. Передача `address: null` — сброс всего объекта целиком.

### 1.5 Retry логика

**Общее правило:** повторяются **только** `5xx` и `429` (с уважением к `Retry-After`). Любой `4xx` — повторять бесполезно: отправитель фиксирует событие как `failed`, шлёт алерт и ждёт правки на стороне получателя. Конкретный фоновый инструмент (очередь, планировщик) — внутренняя деталь стороны и в контракт не выносится.

**1С → Site** (на `5xx` / `429` от сайта):

```
Попытка 1:  сразу
Попытка 2:  через 1 минуту
Попытка 3:  через 5 минут
Попытка 4:  через 30 минут
Попытка 5:  через 2 часа
После 5 попыток → алерт, задача в ручной разбор
```

**Site → 1С** (на `5xx` / `429` от 1С):

```
Попытка 1:  сразу
Попытка 2:  через 1 минуту
Попытка 3:  через 5 минут
Попытка 4:  через 30 минут
Попытка 5:  через 2 часа
(далее — одна попытка в сутки до ручного вмешательства либо отмены)
После 5 попыток → статус исходящего события = "failed", алерт
```

### 1.6 Коды ответов


| Код   | Значение                                                                 |
| ----- | ------------------------------------------------------------------------ |
| `200` | Успешно обновлено **или** идемпотентный повтор «создания»                |
| `201` | Успешно создано (впервые)                                                |
| `202` | Принято к асинхронной обработке (результат прилетит отдельным событием)  |
| `400` | Ошибка валидации входных данных                                          |
| `401` | Неверный или отсутствующий API ключ                                      |
| `404` | Объект не найден                                                         |
| `409` | Конфликт бизнес-правила (не путать с идемпотентным повтором — тот — 200) |
| `429` | Превышен rate limit; повтор по `Retry-After`, далее по §1.5              |
| `500` | Внутренняя ошибка сервера (у получателя), ретраить по §1.5               |


**Унификация создания (идемпотентность):**

- Первое успешное создание объекта → `201` с телом `{ "message": "...", "created": true }`.
- Повтор того же события (по ключу идемпотентности из §1.4) → `200` с телом `{ "message": "Already exists.", "created": false, "idempotent": true }`.
- `409` используется **только** для реального бизнес-конфликта (например, тот же `contract_id` с несовместимыми данными), а не для штатного идемпотентного повтора.

## 2. User Flows — корпоративный кабинет

### Флоу 1 — менеджер регистрирует через 1С

```
Менеджер в 1С создаёт пользователя
        ↓
1С → Site:  user.create (event)
        ↓
Site: создаёт пользователя, шлёт magic-link на email
        ↓
Пользователь переходит по ссылке → попадает в кабинет
```

### Флоу 2 — пользователь регистрируется на сайте сам

Единый исходящий эндпоинт для всех переходов состояния `user × company` — `POST /corporate/users/sync/` (§4.3). Флаги в теле (`registered: true` / `linked: true`) указывают, какой именно переход произошёл; один запрос может отражать сразу оба.

```
Шаг 1: Самостоятельная регистрация (без привязки к компании)
─────────────────────────────────────────────────────
Пользователь заполняет форму на сайте:
  email (REQUIRED), ФИО, телефон
        ↓
Site → 1С: POST /corporate/users/sync/  { registered: true, linked: false, ... }
        ↓
Пользователь получает подтверждение email

Шаг 2: Приглашение в компанию
─────────────────────────────────────────────────────
Вариант А: мастер приглашает через интерфейс сайта
  Мастер вводит email → Site проверяет:
    ├── пользователь существует →
    │     добавляет в компанию сразу
    │     Site → 1С: /corporate/users/sync/  { registered: false, linked: true, login, cabinet_id }
    │
    └── пользователь НЕ существует →
          Site шлёт magic-link приглашение на email
          пользователь регистрируется по ссылке →
          автоматически добавляется в компанию
          Site → 1С: /corporate/users/sync/  { registered: true, linked: true, login, cabinet_id }
                     (один вызов вместо двух отдельных)

Вариант Б: менеджер приглашает через 1С
  1С → Site:  corporate.user.invite (event)
        ↓
  Site проверяет:
    ├── пользователь существует → добавляет в компанию
    └── пользователь НЕ существует → шлёт magic-link
```
