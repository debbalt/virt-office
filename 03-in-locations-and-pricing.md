> **§3.7–3.8** и связанный **OUT §4.6** — объекты аренды, пакеты, прайс и список организаций клиента (GET).

### 3.7 Справочник адресов / объектов аренды

В 1С это справочник **«Объекты аренды»**. Одна запись = одно физическое помещение = одновременно:

- **объект аренды** в каталоге сайта («Адреси Реєстрації Для Вашої Компанії»),
- **место хранения писем** в кабинете клиента (см. §3.4),
- **юридический адрес** для договоров и счетов (см. §3.3).

В будущем ожидается несколько городов (Київ, Львів, Івано-Франківськ) и десятки адресов внутри города.

**Модель владения:** мастер-данные ведёт **1С**. Сайт хранит реплику и **дополняет её в админке** (фото, описания, теги, переводы) — но любые поля, которые приходят из 1С, в админке **read-only**.

**Что приходит из 1С (структурные данные):**

- стабильный `location_id` (= код объекта в 1С);
- город / район — для фильтрации в каталоге;
- два варианта адреса: **публичный** (для каталога) и **точный** (для договоров и счетов);
- **тип пакета** (`package_id`) — к какому пакету услуг относится этот объект;
- флаги активности.

**Что НЕ приходит из 1С (заполняется в админке сайта):**

- название/слоган объекта на любом языке (в 1С — только украинский, на сайте нужны переводы);
- фотогалерея;
- маркетинговое описание объекта («Нежитлове приміщення, цокольний поверх. Секретар щодня...»);
- список «Що включено в оренду» (фичи объекта);
- бейджи/теги: «Бізнес-центр», «Для складу», «Від власника», «NEW» и пр.;
- площадь (`м²`) — присутствует в карточке 1С, но по решению заказчика ведётся в админке сайта;
- координаты для карты (геокодируются на сайте по `address.exact` либо вводятся вручную);
- контактные `phone` / `email` офиса.

**Использование `location_id` в других событиях:**

- `mail.create` / `mail.update` (§3.4) — поле `storage_location_id` опциональное; неизвестный id сайт сохраняет, но игнорирует (мягкое поведение, без `404`).
- `pricing.upsert` (§3.8) — записи ссылаются на тот же `location_id` как на «услугу аренды»; неизвестные id принимаются как pending и активируются после прихода `storage_location.upsert`.

#### `storage_location.upsert` — создать или обновить объект аренды

Семантика — upsert по `location_id`: впервые → создаёт, повторно с теми же полями → идемпотентный `200`, повторно с изменениями → обновление записи.

```json
// Запрос от 1С
{
  "event_type": "storage_location.upsert",
  "payload": {
    "location_id": "000000263",                            // REQUIRED — стабильный код объекта в 1С (как в карточке справочника)
    "city":        "Київ",                                 // REQUIRED — город (для фильтра в каталоге)
    "district":    "Голосіївський",                        // optional — район; null/отсутствие = город без района
    "address": {                                            // REQUIRED — оба варианта адреса (см. §1.3)
      "public": "вул. Лебедєва Академіка, 1, оф. 33",       // REQUIRED — для каталога; без индекса/города
      "exact":  "03143, м. Київ, Голосіївський р-н, вул. Лебедєва Академіка, буд. 1 корп.6, оф. 33"  // REQUIRED — для договоров и счетов
    },
    "package_id":  "vip",                                  // REQUIRED — ссылка на §3.8 / package.upsert; ровно один пакет на объект
    "is_active":   true                                    // optional (default true) — отображается ли в каталоге
  }
}

// Ответ — создан
HTTP 201
{ "message": "Storage location created.", "location_id": "000000263", "created": true }

// Ответ — обновлён или идемпотентный повтор
HTTP 200
{ "message": "Storage location updated.", "location_id": "000000263", "created": false, "idempotent": true }

// Ответ — ошибка валидации
HTTP 400
{
  "error":   "invalid_address",
  "message": "Both address.public and address.exact are required",
  "fields":  ["address.public", "address.exact"]
}
```

**Правила жизненного цикла:**

- **Деактивация** объекта — `storage_location.upsert` с `is_active: false`. Запись остаётся (на неё могут ссылаться архивные письма и активные договоры). В каталоге сайта объект скрывается; в карточках уже арендующих клиентов — отображается как раньше.
- **Повторная активация** — тот же upsert с `is_active: true`.
- **Физическое удаление** объекта из справочника **не поддерживается** этим контрактом.
- **Смена типа пакета** (`package_id`) — допустима через обычный upsert; сайт пересобирает витрину автоматически. Уже активные договоры на этот объект остаются на старых условиях (договор хранит снимок цены/пакета у себя — см. §3.3).
- **Изменение адреса** — через обычный upsert; все ссылки по `location_id` остаются валидными. Активные договоры адрес у себя **не меняют** автоматически (адрес в договоре зафиксирован на момент подписания).

### 3.8 Прайс-лист и справочник пакетов

Цены на услуги аренды виртуального офиса ведёт **1С**. Сайт хранит только реплику и использует её для расчёта стоимости в каталоге, в карточке адреса и при оформлении договора. Цены **не редактируются** в админке сайта.

**Принципы (общие для §3.7 и §3.8):**

- **Никаких пользовательских текстов** из 1С. Названия пакетов («Стандарт», «ВІП-преміум 1»), описания, фичи и переводы — ведутся в **админке сайта** по `package_id`. Из 1С приходят только структурные данные и стабильные коды.
- **Никаких маркетинговых флагов** из 1С: бейджи «Популярно» / «Популярний», теги объектов («Бізнес-центр», «Для складу», «Від власника», «NEW»), порядок отображения пакетов на витрине, группировка карточек на главной — всё в админке сайта.
- **Категории контрагента в 1С:** `Клиент` / `Дисконт` / `Партнер`. На сайт идёт **только `client`** — это и есть «суммы від» в каталоге. Остальные категории внутри 1С.
- **Валюта** — UAH (единственная, см. §1.3). Отдельным полем не передаётся.
- **Связь сущностей:**
  - объект аренды (`storage_location_id`, §3.7) → ровно один `package_id`;
  - документ цен (`pricing.upsert`) → одна комбинация (`customer_category` × `package_id` × диапазон месяцев) и список объектов с ценами;
  - один и тот же `storage_location_id` может фигурировать только в тех документах цен, где `package_id` совпадает с пакетом этого объекта.
- **Длительность аренды** — целое число месяцев. Цена за 1 месяц **зависит от длительности**. В 1С это оформлено диапазонами `Кол-во месяцев аренды: от … до …` (типичные на момент написания контракта: `6–11`, `12–23`, `24–34`, `35–35`); контракт **не фиксирует** конкретные диапазоны — они приходят из 1С как есть.
- **Документ цен 1С** (`Установка цен на услуги аренды`) — атомарная единица передачи: один документ → одно событие `pricing.upsert`.
- **Snapshot-семантика:** `entries[]` — полный список объектов этого документа на стороне 1С. Повторное событие с тем же `pricing_document_id` → **полная замена** содержимого. Объекты, отсутствующие в новом `entries[]`, считаются удалёнными из этого документа.
- **Отображение на сайте** (вычисляет сайт; в контракт не входит):
  - **Главная** — `MIN(price_per_month_after_discount)` по всем активным записям → «від X грн / міс»; год = `X × 12`.
  - **Каталог**  — карточка объекта показывает «від X грн» (минимум по всем длительностям этого объекта) и список тегов из админки.
  - **Карточка объекта** — таблица «Оберіть Строк Оренди» с **ценой за весь срок** = `price_per_month_after_discount × месяцы` и отображением скидки `discount_percent` для каждой строки.

**Зависимости и порядок:** `pricing.upsert` ссылается на `package_id` и `location_id`. Если соответствующий пакет или объект сайту ещё неизвестен, строка прайса принимается, но помечается как `pending` до прихода связанной мастер-записи. Мягкое поведение (без `404`), чтобы рассинхрон справочников не ломал поток.

#### `package.upsert` — справочник типов пакетов аренды

Реестр стабильных кодов пакетов из 1С (Субаренда, Стандарт, ВІП-преміум 1, ВІП и т.д.). Из 1С приходит **только сам факт существования пакета и его активность**. Всё остальное — на стороне сайта:

- название/описание/переводы и иконка пакета,
- порядок отображения, бейджи «Популярно» / «Хіт» / «-25%»,
- маркетинговые теги объектов («Бізнес-центр», «Для складу», «Від власника», «NEW»),
- группировка пакетов на витрине.

```json
// Запрос от 1С
{
  "event_type": "package.upsert",
  "payload": {
    "package_id": "standard",     // REQUIRED — стабильный код пакета (slug / внутренний id 1С); по нему сайт находит свой контент
    "is_active":  true            // optional (default true) — участвует ли в каталоге; деактивация скрывает пакет на витрине
  }
}

// Ответ — создан
HTTP 201
{ "message": "Package created.", "package_id": "standard", "created": true }

// Ответ — обновлён или идемпотентный повтор
HTTP 200
{ "message": "Package updated.", "package_id": "standard", "created": false, "idempotent": true }
```

**Правила:**

- **Неизвестный `package_id` в админке** — сайт принимает событие и заводит «пустую» запись пакета (без названия/описания); в публичный каталог она не выводится, пока контент-менеджер не заполнит её. В админке появляется уведомление «новый пакет из 1С, требуется наполнение».
- **Деактивация** пакета — `is_active: false`. Существующие записи прайса и объекты с этим пакетом сохраняются, но из публичного каталога пакет уходит.
- **Физическое удаление** не поддерживается.

#### `pricing.upsert` — снимок прайс-листа из документа 1С

Семантика — upsert по `pricing_document_id` (внутренний id документа в 1С). Повтор без изменений → `200 idempotent`. Повтор с любыми изменениями полей или `entries[]` → **полная замена** содержимого документа на сайте.

```json
// Запрос от 1С
{
  "event_type": "pricing.upsert",
  "payload": {
    "pricing_document_id":     "aar-130",                      // REQUIRED — уникальный id документа в 1С (idempotency key)
    "pricing_document_number": "000000130",                    // optional — человекочитаемый номер (для аудита/логов)
    "pricing_document_date":   "2026-04-13T14:40:42Z",         // REQUIRED — дата документа в 1С
    "effective_from":          "2026-04-13",                   // REQUIRED — дата вступления цен в силу (YYYY-MM-DD)
    "effective_to":            null,                           // optional, null = бессрочно; если задана — дата окончания действия
    "customer_category":       "client",                       // REQUIRED — закрытый enum; на сайт идёт только "client"
    "package_id":              "standard",                     // REQUIRED — ссылка на §3.8 / package.upsert
    "months_from":             12,                             // REQUIRED — нижняя граница диапазона длительности (включительно), целое >= 1
    "months_to":               23,                             // REQUIRED — верхняя граница диапазона (включительно), >= months_from
    "entries": [                                               // REQUIRED — полный список объектов в документе (snapshot)
      {
        "storage_location_id":            "000000263",                // REQUIRED — id объекта аренды (= location_id из §3.7; то же имя поля, что и в mail.*)
        "price_per_month":                950.00,                     // REQUIRED — базовая цена за 1 мес (до скидки), число >= 0, 2 знака
        "discount_percent":               45.263,                     // optional, default 0; диапазон [0..100], до 3 знаков
        "price_per_month_after_discount": 520.00                      // REQUIRED — финальная цена за 1 мес (после скидки); сайт использует её для расчётов
      }
      // ... остальные объекты документа
    ]
  }
}

// Ответ — создан
HTTP 201
{
  "message":              "Pricing document created.",
  "pricing_document_id":  "aar-130",
  "entries_accepted":     17,
  "entries_pending":      0,                                   // > 0, если часть storage_location_id ещё неизвестны сайту
  "created":              true
}

// Ответ — обновлён или идемпотентный повтор
HTTP 200
{
  "message":              "Pricing document updated.",
  "pricing_document_id":  "aar-130",
  "entries_accepted":     17,
  "entries_pending":      0,
  "created":              false,
  "idempotent":           true
}

// Ответ — ошибка валидации
HTTP 400
{
  "error":   "invalid_pricing",
  "message": "months_to must be >= months_from",
  "fields":  ["months_from", "months_to"]
}
```

**Правила и ограничения:**

- `customer_category` — закрытый enum: `"client"`, `"discount"`, `"partner"`. **На сайт идёт только `"client"`** — это публичные цены. Дисконтные/партнёрские категории остаются внутри 1С и в этом контракте не описаны (если 1С пришлёт их — сайт принимает событие, но в публичный каталог не выводит).
- `months_from` / `months_to` — целые, `>= 1`, `months_to >= months_from`. Совпадение (`from == to`) допустимо (как для диапазона `35-35`).
- **Согласованность пакета:** все `storage_location_id` в `entries[]` должны принадлежать тому же `package_id`, что указан в документе (см. §3.7 — у каждого объекта ровно один пакет). Несоответствие → запись принимается, но строка попадает в `entries_pending` и в админке появляется warning «несоответствие пакета объекту».
- В пределах одного `pricing_document_id` `storage_location_id` в `entries[]` должен быть **уникален** — иначе `400 invalid_pricing`.
- **Удаление документа** (например, ошибочно созданного в 1С): отдельным событием `pricing.delete` с `pricing_document_id`. Сайт убирает все записи этого документа из активного прайса (тело документа не хранит).

#### `pricing.delete` — удалить документ цен

```json
{
  "event_type": "pricing.delete",
  "payload": {
    "pricing_document_id": "aar-130"
  }
}

// Ответ — удалён
HTTP 200
{ "message": "Pricing document deleted.", "pricing_document_id": "aar-130", "deleted": true }

// Ответ — документа нет (идемпотентно)
HTTP 200
{ "message": "Pricing document not found, nothing to delete.", "pricing_document_id": "aar-130", "deleted": false, "idempotent": true }
```

> Замечание: повторная отправка `pricing.delete` для уже удалённого документа — идемпотентна и не считается ошибкой. Восстановление возможно через повторный `pricing.upsert` с тем же `pricing_document_id`.
