> **§3.4** и связанные **OUT §4.4, §4.7** — письма и доставка (заказ скана, ТТН и статусы НП → 1С).

### 3.4 Письма

#### `mail.create` — новое письмо

```json
// Запрос от 1С
{
  "event_type": "mail.create",
  "payload": {
    "message_id":          "msg-001",                                          // REQUIRED
    "login":               "01/02/03",                                         // REQUIRED — код клиента в 1С
    "scan_date":           "2026-03-20T10:00:00Z",                             // REQUIRED
    "attachment_link":     "https://storage.virtoffice.ua/scans/msg-001.pdf",  // REQUIRED
    "issue_date":          "2026-03-18",
    "storage_location_id": "kyiv-main"                                         // optional — где физически хранится письмо (см. §3.7); если не передано — сайт отображает стандартный адрес
  }
}

// Ответ — успех
HTTP 201
{ "message": "Record successfully created." }

// Ответ — контрагент (клиент по login) не найден на сайте
HTTP 404
{
  "error": "Client not found",
  "login": "01/02/03"
}
```

#### `mail.update` — обновить письмо

Применяется для **исправления ошибок менеджера** в карточке письма (опечатка в дате, неправильно прикреплённый скан и т.п.). Уведомление клиенту в этом случае **не** генерируется: исправление — внутренняя операция, не новый бизнес-повод. Готовность скана как отдельный бизнес-процесс — через `mail.scan.ready`, там уведомление создаётся (§3.6, `mail.scan_ready`).

```json
// Запрос от 1С
{
  "event_type": "mail.update",
  "payload": {
    "message_id":          "msg-001",          // REQUIRED
    "storage_location_id": "lviv-central",     // optional — изменение места хранения (переезд письма между офисами)
    "scan_date":           "2026-03-21T10:00:00Z",
    "attachment_link":     "https://storage.virtoffice.ua/scans/msg-001-v2.pdf",
    "issue_date":          "2026-03-19"
  }
}

// Ответ — успех
HTTP 200
{ "message": "Record successfully updated." }
```

#### `mail.status.update` — обновить статус письма

```json
// Запрос от 1С
{
  "event_type": "mail.status.update",
  "payload": {
    "message_id":  "msg-001",               // REQUIRED
    "status":      "issued",                // REQUIRED
    "issued_to":   "Директор Иваненко",
    "issued_date": "2026-03-22T14:30:00Z"
  }
}

// Статусы (базовый enum; см. матрицу ниже и ТЗ §4.1.1):
// "received"  — получено на склад / новое поступление (ТЗ: Новое)
// "scanned"   — отсканировано
// "pending"   — ожидает выдачи / этап до НП (см. матрицу: «Заказана доставка»)
// "issued"    — выдано клиенту (ТЗ: Выдано в офисе)
// "delivery"  — отправлено НП (ТЗ: Отправлено НП)
// "disposed"  — утилизировано (ТЗ: Утилизировано) — расширение enum, согласовать с 1С
// "archived"  — архивировано (ТЗ: Архивировано) — расширение enum, согласовать с 1С

// Ответ — успех
HTTP 200
{ "message": "Status updated." }
```

#### `delivery.ttn.manual` — ТТН «Новой почты» внесена в 1С вручную (менеджер)

Сценарий: в учёте 1С менеджер вручную фиксирует номер ТТН для отправки письма; данные должны отразиться в кабинете на сайте.

**Поведение сайта после приёма события:**

- Сохранить `ttn_number` (и опциональные получателя) в карточке письма. Если данные кабинета расходятся с `payload`, **применяются данные из 1С** (источник истины для отображения и дальнейшей логики — входящее событие).
- Считать доставку по этому письму **уже оформленной**: **повторно заказать доставку** посылки в кабинете **нельзя** (тот же принцип, что при ТТН, созданной через API НП из кабинета).
- **Автоматически** запросить у официального **API Новой почты** актуальные данные по `ttn_number`. Для базового отслеживания методом `TrackingDocument.getStatusDocuments` минимально достаточно пары `DocumentNumber` (ТТН) + `Phone` (телефон отправителя или получателя — первый известный). 
- Далее цепочка статусов — как в §4.7: **НП → сайт → 1С** через `POST /delivery/status/`.

> **Важно про уникальность `ttn_number`.** Один номер ТТН Новой почты может соответствовать **нескольким письмам** одного контрагента: когда клиент заказывает доставку «всего что накопилось», на партию писем оформляется одна ТТН. В модели данных сайта (и в логике идемпотентности из §1.4) `ttn_number` — **не** уникальный ключ письма; связка «письмо ↔ ТТН» — это отношение `M:1`, ключ письма остаётся `message_id`. Сайт при `delivery.ttn.manual` **накапливает** привязки писем к одной ТТН, а не затирает предыдущие.

```json
// Запрос от 1С
{
  "event_type": "delivery.ttn.manual",
  "payload": {
    "message_id":        "msg-001",           // REQUIRED
    "login":             "01/02/03",          // REQUIRED — код контрагента в 1С
    "ttn_number":        "20450000000001",    // REQUIRED — номер ТТН НП
    "recipient_name":    "Иваненко Иван",     // optional — если известен в 1С
    "recipient_phone":   "+380991234567"      // optional — если известен в 1С
  }
}

// Ответ — успех
HTTP 200
{ "message": "TTN applied from 1C." }

// Ответ — письмо или контрагент не найдены
HTTP 404
{
  "error": "Mail or counterparty not found",
  "message_id": "msg-001",
  "login": "01/02/03"
}
```

> Идемпотентность: см. §1.4 (`message_id` + `ttn_number`). Повтор с тем же набором — успех без повторного побочного эффекта. Повтор с **другим** `ttn_number` для того же `message_id` — применить новые данные из 1С и переключить синхронизацию с API НП на новый номер.

#### Матрица статусов письма: ТЗ ↔ API ↔ источник

Ниже — соответствие **ТЗ** (п. 4.1.1 «статус письма», п. 4.1.2 «статус сканирования») полям и событиям контракта. Значения `status` в `mail.status.update`.

**Статус письма** 


| Статус в ТЗ               | `status` в `mail.status.update`         | Кто задаёт истину                  | События / примечание                                                                                                                                                                  |
| ------------------------- | --------------------------------------- | ---------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Новое (поступление из 1С) | `received`                              | 1С                                 | Первичное поступление: `mail.create`; при необходимости явное `mail.status.update`                                                                                                    |
| Заказана доставка         | `pending` (ожидает выдачи / этап до НП) | Сайт → 1С OUT **или** 1С → Site IN | После `POST /delivery/ttn/` **или** `delivery.ttn.manual` (ручная ТТН менеджером); дубль ТТН запрещён (ТЗ п. 4.3); при привязанной ТТН повторный заказ доставки в кабинете недоступен |
| Отправлено НП             | `delivery`                              | НП → сайт → 1С                     | После данных из API НП (в т.ч. по ТТН из кабинета **или** внесённой вручную в 1С) сайт обновляет письмо и шлёт в 1С **OUT** `POST /delivery/status/` (см. §4.7)                       |
| Доставлено НП             | *(не дублировать в IN)*                 | API НП → сайт → 1С                 | Истина — **OUT** `POST /delivery/status/` (`status`, например `delivered`); опционально сайт обновляет внутренний статус письма без отдельного `mail.status.update` от 1С             |
| Выдано в офисе            | `issued`                                | 1С                                 | `mail.status.update` + `issued_to` / `issued_date`                                                                                                                                    |
| Утилизировано             | `disposed`                              | 1С                                 | Согласовать с 1С перед релизом                                                                                                                                                        |
| Архивировано              | `archived`                              | 1С                                 | Согласовать с 1С перед релизом                                                                                                                                                        |

## OUT — Site → 1С (письма: заказ скана; Новая почта — ТТН и статусы)

### 4.4 Запрос сканирования письма (сайт → 1С)

Триггер: клиент нажал «Заказать скан» в кабинете. Сайт фиксирует состояние и передаёт заявку в 1С для учёта и передачи в сервис-отдел.

```http
POST https://mock.1c.virtoffice.ua/api/v2/mails/scan-request/
X-API-Key: {api_key}
Content-Type: application/json
```


| Поле           | Обяз. | Описание                                                            |
| -------------- | ----- | ------------------------------------------------------------------- |
| `message_id`   | да    | Идентификатор письма (как в `mail.create`)                          |
| `login`        | да    | Код клиента в 1С (как в `mail.create`)                              |
| `request_id`   | нет   | UUID заявки на стороне сайта; **рекомендуется** для идемпотентности |
| `requested_at` | нет   | Время действия пользователя, ISO 8601                               |


**Идемпотентность:** повтор с тем же `request_id` → **200**, без второй заявки. Без `request_id` — дедупликация по `message_id` (второй запрос → **200** «уже зарегистрировано») либо **409** — зафиксировать с командой 1С.

```jsonc
{
  "message_id": "msg-001",
  "login": "01/02/03",
  "request_id": "660e8400-e29b-41d4-a716-446655440001",
  "requested_at": "2026-03-20T12:00:00Z"
}
```

**Ответ — заявка принята (HTTP 201):**

```json
{
  "message": "Scan request accepted.",
  "request_id": "660e8400-e29b-41d4-a716-446655440001"
}
```

**Ответ — идемпотентный повтор (HTTP 200):**

```json
{
  "message": "Scan request already registered.",
  "duplicate": true
}
```

**Ответ — письмо не найдено (HTTP 404):**

```json
{
  "error": "Message not found",
  "message_id": "msg-001"
}
```

**Ответ — операция недопустима для текущего статуса (HTTP 409):**

```json
{
  "error": "Scan request not allowed for current mail state",
  "message_id": "msg-001"
}
```

> После **201** на сайте выставляется статус сканирования «Скан заказан» (ТЗ §4.1.2); готовый файл — через IN `mail.scan.ready` / `mail.update`.

### 4.7 Доставка «Новая почта»: API НП на сайте, уведомление 1С

**Flow A — доставка из кабинета:**

1. Клиент оформляет доставку в кабинете.
2. **Сайт** вызывает **официальное API Новой почты** (справочники города/отделения, создание ЕН/ТТН и т.д.) — `city_ref`, `warehouse_ref` и другие поля используются **в запросах к НП**, не дублируются в контракте с 1С.
3. После **успешного** ответа НП (номер ТТН получен) сайт **асинхронно** уведомляет **1С**: `POST /delivery/ttn/` — чтобы в учёте были ТТН и связь с письмом/контрагентом.
4. Дальнейшие **изменения статуса** отправления сайт получает от НП (webhook, polling или смешанная схема — реализация) и при каждой существенной смене **уведомляет 1С**: `POST /delivery/status/`.

**Flow B — ТТН внесена в 1С вручную (менеджер):**

1. Менеджер вносит номер ТТН в 1С.
2. **1С → Site:** `delivery.ttn.manual` (см. §3.4). Сайт применяет данные; при конфликте с кабинетом **главенствуют данные из 1С**; повторный заказ доставки по письму **недоступен**.
3. **Сайт** по полученному `ttn_number` **автоматически** запрашивает данные у **API Новой почты** и далее ведёт ту же цепочку статусов, что в Flow A: **НП → сайт → 1С** через `POST /delivery/status/`.

**1С не вызывает** API Новой почты ни при оформлении из кабинета, ни при ручном вводе ТТН; запросы к API НП выполняет **сайт**. Источник нормализованных статусов доставки в 1С — по-прежнему **НП → сайт → 1С** (OUT).

#### 4.7.1 Уведомление 1С о созданной ТТН

Вызывается **после** успешного создания отправления в API НП. Одна ТТН может покрывать **несколько писем** одного контрагента (batch-доставка), поэтому связь с письмами передаётся массивом.

**Инициатор заказа (контекст UI):** если доставку оформили из **корпоративного** кабинета, в теле **обязательно** передаются **`cabinet_id`** и **`initiated_by_email`**: это идентификация кабинета и **email пользователя** (мастера или сотрудника с membership и доступом к компании), который выполнил действие «Заказать доставку» в интерфейсе сайта. Для заказа из **индивидуального** кабинета компании эти поля **не передаются** — считается, что инициатором является сессия владельца записи по **`login`** без дополнительной связки membership.

Поля **`cabinet_id`** и **`initiated_by_email`** в связке используются только для аудита и учёта в 1С; ключ идемпотентности запроса по-прежнему см. §1.4 (`ttn_number` и состав **`message_ids`**).

```http
POST https://mock.1c.virtoffice.ua/api/v2/delivery/ttn/
X-API-Key: {api_key}
Content-Type: application/json
```

```jsonc
{
  "ttn_number": "20450000000001",
  "login": "01/02/03",
  "message_ids": ["msg-001", "msg-002"],
  "recipient_name": "Иваненко Иван",
  "recipient_phone": "+380991234567",
  "cabinet_id": "cab-001",
  "initiated_by_email": "accountant@company.ua"
}
```

Обязательные всегда: `ttn_number`, `login`, `message_ids`, `recipient_name`, `recipient_phone`. **`cabinet_id`** и **`initiated_by_email`** — см. правило выше (оба REQUIRED при заказе из корп. кабинета; при заказе из индивидуального — полей нет в теле). Передать только одно из двух или указать `cabinet_id` без `initiated_by_email` — **`400`** (`invalid_delivery_context`).

Пример для **индивидуального** кабинета (`cabinet_id` и `initiated_by_email` не передаются):

```jsonc
{
  "ttn_number": "20450000000002",
  "login": "01/02/03",
  "message_ids": ["msg-010"],
  "recipient_name": "Иваненко Иван",
  "recipient_phone": "+380991234567"
}
```

**Ответ — успех (HTTP 201):**

```json
{ "message": "TTN registered in 1C." }
```

**Ответ — идемпотентный повтор (HTTP 200):**

```json
{ "message": "TTN already registered.", "idempotent": true }
```

#### 4.7.2 Уведомление 1С об изменении статуса доставки

Вызывается сайтом, когда из API НП известен новый статус отправления. Дедупликация на стороне 1С — по `ttn_number + delivery_status` (см. §1.4).

```http
POST https://mock.1c.virtoffice.ua/api/v2/delivery/status/
X-API-Key: {api_key}
Content-Type: application/json
```

```jsonc
{
  "ttn_number": "20450000000001",
  "delivery_status": "delivered",
  "np_status_code": "9",
  "np_status_text": "Відправлення отримано",
  "message_ids": ["msg-001", "msg-002"],
  "login": "01/02/03",
  "status_changed_at": "2026-03-22T14:00:00Z"
}
```

Поля: `ttn_number`, `delivery_status`, `status_changed_at` — REQUIRED; `np_status_code`, `np_status_text`, `message_ids`, `login` — optional (см. таблицу enum ниже).

**Закрытый enum `delivery_status`** (маппинг на группы статусов НП — на стороне сайта, 1С получает уже нормализованное значение):


| `delivery_status`        | Что это                                                      | Примеры групп/статусов НП (для справки)                                                          |
| ------------------------ | ------------------------------------------------------------ | ------------------------------------------------------------------------------------------------ |
| `created`                | ЭН оформлена, ещё не передана к отправке                     | «Відправник оформив посилку, але ще не відправив»                                                |
| `accepted`               | Прийнято відділенням / забрано кур'єром                      | «Прийнята у відділенні», «Прийнята кур'єром»                                                     |
| `in_transit`             | У дорозі (між пунктами / в депо / на проміжному складі)      | «Посилка в дорозі», «Прибула в депо», «ArrivalSC», «Виїхала з проміжного пункту»                 |
| `arrived_destination`    | Прибула до міста/відділення отримання                        | «Прибула до міста доставки»                                                                      |
| `out_for_delivery`       | Кур'єр вирушив до отримувача                                 | «Кур'єр вирушив до отримувача»                                                                   |
| `delivered`              | Отримано отримувачем                                         | «Отримано», «Отримано у відділенні»                                                              |
| `returning`              | Повертається відправнику (відмова / не забрано)              | «Отримувач відмовився від посилки. Відправнику повідомлено»                                      |
| `returned_to_sender`     | Повернуто відправнику (фінальний стан повернення)            | те саме, фінальна стадія повернення                                                              |
| `customs`                | Митний контроль / оформлення                                 | «Готується до передачі на митницю», «В процесі митного оформлення», «Митне оформлення завершено» |
| `transferred_to_partner` | Передано партнеру для подальшої доставки                     | «TransferToPartner», «Передано партнеру GoFo Express»                                            |
| `redirected`             | Замовлено переадресування або перенесення дати доставки      | «Замовлено переадресування», «Клієнт замовив перенесення дати доставки»                          |
| `not_found`              | Номер ТТН не знайдено в API НП (тимчасова/постійна проблема) | «Номер не найден»                                                                                |


- `delivery_status` расширяется **только** по согласованию с 1С (контрактное изменение). Сырые поля `np_status_code` / `np_status_text` позволяют 1С сохранить точное значение НП для диагностики без расширения enum.
- Сайт шлёт событие на **каждое** значимое изменение `delivery_status`; промежуточные движения внутри одной группы (разные точки маршрута внутри `in_transit`) могут не транслироваться, но текст/код последнего известного состояния — в `np_status_text` / `np_status_code`.

**Ответ — успех (HTTP 200):**

```json
{ "message": "Delivery status synced to 1C." }
```
